Card Payment Processing System

Five paradigms. One system. One programming model.

The challenge

A card payment touches multiple processing styles across its lifecycle — synchronous calls, event processing, long-running workflows, batch jobs, and real-time queries.

Today, each paradigm typically requires its own infrastructure, its own failure handling, and its own operational model. What if they didn't?

Five paradigms, one system

  • 1 Stateful RPC — Synchronous payment authorization
  • 2 Event-Driven Processing — Clearing file fan-out into parallel processing
  • 3 Per-Event Workflows — Durable per-payment orchestration
  • 4 Batch Processing — End-of-day settlement across all merchants
  • 5 Real-Time Serving — Live merchant reconciliation queries

Architecture

Demo walkthrough

  • Authorize paymentsThree stateful RPCs, exclusive per-key access
  • Send clearing fileFan-out via send() — fire-and-forget into durable log
  • Per-event workflowsAuth check, debit, accumulate — with automatic retries
  • Query reconciliationConcurrent reads on live state, no locking
  • Trigger settlementParallel fan-out with DurableFuture.all()
  • IdempotencyRe-send clearing file — nothing changes
  • Late authorizationWorkflow suspends, auth resolves it — no polling

Key takeaways

Durable execution

Transient failures are handled automatically. Side effects are journaled and retried. No manual error handling.

One infrastructure

RPCs, event processing, workflows, batch, and serving — all in Restate. No Kafka, no external workflow engine, no cache layer.

Exactly-once semantics

Workflows keyed by payment ID. Re-sending the same file is a no-op. Idempotency with zero extra code.

Just code

No YAML, no DSLs, no state machine config. Regular Java with Restate annotations. Easy to test, easy to debug.
